# 1.4 — Modelo de Datos por Aplicación

> **Tipo B — Plataforma de Delivery**

Para cada webapp, describir las entidades principales de su base de datos: tablas, campos relevantes y relaciones. No es necesario un DER formal, pero sí que quede claro qué persiste cada app.

También identificar posibles duplicados entre apps (ej: usuarios) y definir cómo se resuelven las inconsistencias.

## Buyer App

### Entidades principales

### 1. `users` (Perfiles)
Almacena la información base del comprador vinculada a la identidad centralizada de Clerk.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `client_id` | `String` (PK) | ID único de usuario proveniente de Clerk (campo sub). |
| `name` | `String` (Unique) | Nombre completo del usuario. |
| `email` | `String` | Correo electrónico gestionado por Clerk. |
| `phone` | `String` | Teléfono de contacto. |

### 2. `addresses` (Direcciones)
Almacena direcciones específicas elegidas por el usuario (casas, oficinas, etc) en las que quiera recibir sus pedidos.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `address_id` | `UUID` (PK) | Identificador de la dirección. |
| `client_id` | `String` (FK) | Relación con el usuario. |
| `title` | `String` | Etiqueta elegida por el usuario(ej: "Casa"). |
| `street` | `String` | Calle y altura. |
| `city` | `String` | Nombre de la ciudad. |
| `lat` | `Float` | Latitud geográfica. |
| `lng` | `Float` | Longitud geográfica. |

### 3. `purchases`(Compra Global)
Representa la transacción financiera única generada por el Seller para el proceso de pago.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `purchase_id` | `UUID` (PK) | Identificador único de la transacción. |
| `client_id` | `String` (FK) |  Referencia al usuario que realiza la compra.|
| `address_id` | `UUID` (FK) | Dirección de entrega elegida. |
| `amount` | `Float` | Monto total a abonar por la suma de todos los sub-pedidos. |
| `status` | `Enum` | Estado del pago: `PENDING`, `PAID`, `CANCELLED`. |

### 4. `Orders`(Paquetes Individuales)
Representa cada paquete independiente que forma parte de la compra global y que posee su propio ciclo logístico.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `order_id` | `UUID` (PK) | Identificador del paquete generado por el Seller. |
| `purchase_id` | `UUID` (FK) |  Relación con la compra global (transacción madre).|
| `store_name` | `String` | Nombre del comercio para visualización en el seguimiento. |
| `status` | `Enum` | Estado logístico: `PAYMENT_PENDING`,`PREPARING`, `ON_THE_WAY`,`OUT_FOR_DELIVERY`, `DELIVERED`,`CANCELLED`. |
| `delivery_code` | `Integer` | Código OTP (Nullable) para validar la entrega con el repartidor. |

### 5. `Order_items`(Detalles de productos)
Desglose de los artículos específicos contenidos en cada paquete individual.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `Serial` (PK) | Identificador del paquete generado por el Seller. |
| `order_id` | `UUID` (FK) |  Relación con el paquete específico.|
| `product_name` | `String` | Nombre del producto al momento de la compra. |
| `quantity` | `Integer` | Cantidad de unidades del producto. |

## Seller App

### Entidades principales

### 1. `stores` (Negocios)
Almacena la información base del negocio vinculada a la identidad centralizada de Clerk.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `store_id` | `Serial` (PK) | Identificador único proveniente de Clerk. |
| `name` | `String` (Unique) | Nombre completo del negocio. |
| `email` | `String` | Correo electrónico de contacto comercial.|
| `category` | `String` | Rubro o categoría (ej: "Pizzería"). |
| `image_url` | `text` | Imágen de la tienda. |

### 2. `addresses` (Direcciones)
Almacena las direcciones físicas de retiro. Es el dato que se envía a Delivery para las cotizaciones y los retiros.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `UUID` (PK) | Identificador de la dirección. |
| `store_id` | `String` (FK) | Relación con el comercio. |
| `street` | `String` | Calle y altura. |
| `city` | `String` | Nombre de la ciudad. |
| `lat` | `Float` | Latitud geográfica. |
| `lng` | `Float` | Longitud geográfica. |

### 3. `products` (Productos)
Productos que se mostrarán en el catálogo de un negocio y se agruparan por paquetes.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `product_id` | `UUID` (PK) | Identificador único del producto. |
| `store_id` | `Integer` (FK) | Referencia a la tienda dueña. |
| `name` | `String` | Nombre del producto. |
| `price` | `Numeric` | Precio unitario actual. |
| `stock` | `Integer` | Cantidad disponible en inventario. |
| `image_url` | `text` | Imágen del producto. |

### 4. `packages` (Paquetes)
Representa la fragmentación del carrito. Cada registro es un paquete independiente con su propio costo logístico.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `package_id` | `UUID` (PK) | ID del paquete individual. |
| `payment_order_id` | `UUID` (FK) | ID de la compra global (proveniente de Payments). Agrupa todos los paquetes de un mismo carrito. |
| `location_id` | `UUID` | Relación con store_locations (de dónde se retira). |
|`location_id`| `UUID` | Dirección del comprador. |
| `shipping_cost` | `Numeric` | Costo de envío cotizado para este paquete. |
| `delivery_id` | `UUID` (FK) | ID de misión devuelto por Delivery App (Nullable). |
| `status` | `Enum` | PREPARING, READY_TO_PICKUP, IN_TRANSIT, DELIVERED. |

### 5. `package_items` (Detalle de productos)
Snapshots de los productos contenidos en cada paquete para el armado en el local y registro histórico de precios.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `BigSerial` (PK) | ID incremental. |
| `package_id` | `UUID` (FK) | Relación con el paquete. |
| `product_name` | `String` | Nombre del producto al momento de la compra. |
| `quantity` | `Integer` | Cantidad de unidades. |
| `price_of_purchase` | `Numeric`  | Precio unitario al momento de la transacción. |

## Delivery App

### Entidades principales
### 1. `deliveries` (Misión Logística / Viaje)
Es la entidad raíz que representa el estado y tiempos de un traslado.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `UUID` (PK) | Identificador único del viaje. |
| `order_id` | `String` (Unique) | Referencia al pedido original (generado por Payments). |
| `status` | `Enum` | `ACCEPTED_FOR_ASSIGNMENT`, `COURIER_ASSIGNED`, `PICKED_UP`, `OUT_FOR_DELIVERY`, `DELIVERED`, `DELIVERY_FAILED`, `CANCELLED_SUCCESSFULLY`. |
| `requested_by_app` | `String` | Aplicación que disparó la misión (ej: `SELLER`). |
| `ready_at` | `Timestamp` | Pedido listo para ser retirado. |
| `assigned_at` | `Timestamp` | Momento de la asignación del repartidor. |
| `picked_up_at` | `Timestamp` | Momento del retiro efectivo en el local. |
| `delivered_at` | `Timestamp` | Momento de la entrega final al comprador. |
| `close_result` | `Enum` | Resultado técnico del cierre: `DELIVERED`, `DELIVERY_FAILED`. |

### 2. `delivery_context_snapshots`
Copia de seguridad de los datos operativos para garantizar autonomía durante el viaje.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `UUID` (PK) | Identificador del snapshot. |
| `delivery_id` | `UUID` (FK) | Relación con la tabla `deliveries`. |
| `seller_id` | `String` | ID del vendedor (**Clerk**). |
| `seller_address` | `String` | Dirección física de retiro. |
| `seller_contact_masked` | `String` | Teléfono de contacto enmascarado. |
| `buyer_id` | `String` | ID del comprador (**Clerk**). |
| `buyer_address` | `String` | Dirección física de entrega. |
| `buyer_contact_masked` | `String` | Teléfono de contacto enmascarado. |
| `otp_required` | `Boolean` | Flag para requerir código de validación al entregar. |

### 3. `couriers` (Repartidores / Vehículos)
Maestro de los operadores logísticos disponibles en la plataforma.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `UUID` (PK) | Identificador interno del repartidor. |
| `clerk_user_id` | `String` (Unique) | ID de usuario gestionado por **Clerk**. |
| `display_name` | `String` | Nombre público del repartidor o vehículo. |
| `vehicle_type` | `String` | Tipo (Moto, Bici, Dron, etc.). |
| `vehicle_plate` | `String` | Patente o identificador del vehículo. |
| `availability_status` | `Enum` | `AVAILABLE`, `ASSIGNED`, `OFFLINE`. |

### 4. `delivery_assignments`
Historial de vinculaciones entre repartidores y misiones.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `UUID` (PK) | Identificador de la asignación. |
| `delivery_id` | `UUID` (FK) | Relación con la misión. |
| `courier_id` | `UUID` (FK) | Relación con el repartidor. |
| `assignment_status` | `Enum` | `ASSIGNED`, `REASSIGNED`, `REJECTED`. |

### 5. `delivery_tracking_points` (Telemetría)
Registro masivo de coordenadas para el trazado de rutas en tiempo real.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `BigSerial` (PK) | ID incremental de alta frecuencia. |
| `delivery_id` | `UUID` (FK) | Relación con el viaje. |
| `lat` | `Numeric` | Coordenada de Latitud. |
| `lng` | `Numeric` | Coordenada de Longitud. |
| `recorded_at` | `Timestamp` | Momento exacto de la captura GPS. |

### 6. `delivery_status_events`
Bitácora de auditoría para el seguimiento de la evolución del pedido.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `BigSerial` (PK) | ID incremental. |
| `delivery_id` | `UUID` (FK) | Referencia al viaje. |
| `to_status` | `Enum` | Estado al que transicionó el pedido. |
| `event_source` | `String` | Origen del evento (`DELIVERY`, `PAYMENTS`, `SELLER`, `BUYER`). |

## Payments App
### Tabla `orders`

Fuente de verdad de cada transacción de compra. Es la única tabla estrictamente requerida en la fase actual del proyecto.

| Columna | Tipo | Descripción |
|---------|------|-------------|
| `order_id` | `uuid` PK | Generado por Payments con `uuid v9`. Es el identificador global que todos los demás servicios referencian. |
| `buyer_id` | `text` NOT NULL | `sub` del JWT de Clerk del comprador. |
| `store_id` | `uuid` NOT NULL | ID del comercio vendedor (snapshot — no FK a Seller). |
| `status` | `text` NOT NULL | Estado financiero de la orden. Ver tabla de estados más abajo. |
| `total_amount` | `numeric(10,2)` NOT NULL | Subtotal de ítems + `delivery_cost`. |
| `delivery_cost` | `numeric(10,2)` NOT NULL | Costo de envío cotizado por Delivery en el momento de la orden. |
| `mp_preference_id` | `text` | ID de la preferencia de MercadoPago (`MP-...`). Se usa para inicializar el Payment Brick. |
| `mp_payment_id` | `bigint` | ID del pago en MercadoPago, disponible tras `POST /api/payments/process`. |
| `items_snapshot` | `jsonb` NOT NULL | Snapshot inmutable del carrito al momento de la compra. Estructura: `[{ product_id, seller_id, name, quantity, unit_price, subtotal }]`. El `seller_id` por ítem es necesario para calcular las liquidaciones individuales en el cierre. |
| `delivery_address_snapshot` | `jsonb` NOT NULL | Snapshot inmutable de la dirección de entrega. Estructura: `{ street, city, zip }`. |
| `delivery_quote_snapshot` | `jsonb` NOT NULL | Snapshot inmutable de la cotización de Delivery. Estructura: `{ quote_id, cost, estimated_minutes }`. |
| `created_at` | `timestamptz` DEFAULT `now()` | Timestamp de creación. |
| `updated_at` | `timestamptz` | Mantenido automáticamente por un trigger de Postgres. |

#### Estados posibles de `status`

| Valor | Descripción |
|-------|-------------|
| `payment_pending` | Orden creada, esperando que el comprador complete el pago. |
| `paid` | Pago aprobado por MercadoPago. Seller recibe notificación para iniciar logística. |
| `failed` | Pago rechazado o fallido. |
| `cancelled` | Orden cancelada (status `cancelled` en MP). |
| `closed` | Entrega confirmada + liquidaciones generadas. |

### Tabla `liquidations` *(implementación futura)*

Pagos salientes a vendedores y repartidores. Se generan cuando Delivery confirma la entrega via `POST /payments/operations/{order_id}/close`.

| Columna | Tipo | Descripción |
|---------|------|-------------|
| `liquidation_id` | `uuid` PK | ID de la liquidación. |
| `order_id` | `uuid` FK → `orders` | Orden asociada. |
| `recipient_type` | `text` | `seller` o `courier`. |
| `recipient_id` | `text` | ID del vendedor o repartidor. |
| `amount` | `numeric(10,2)` | Monto a liquidar. |
| `status` | `text` | `pending` → `paid`. |
| `created_at` | `timestamptz` | Timestamp de creación. |

## Principios de diseño del modelo

| Principio | Aplicación |
|-----------|-----------|
| **Snapshots inmutables** | `items_snapshot`, `delivery_address_snapshot` y `delivery_quote_snapshot` congelan los datos externos al momento de la orden. Cambios posteriores en el catálogo de Seller no afectan órdenes históricas. |
| **`order_id` como UUID fuente de verdad** | Payments genera el `order_id`. Todos los demás servicios (Seller, Delivery, Buyer) lo referencian como identificador de la transacción. |
| **Sin FK cross-service** | `store_id`, `buyer_id` y demás IDs de otros servicios se almacenan como `text` o `uuid` simples, sin FK a bases de datos externas. La integridad referencial se garantiza via contratos de API. |
| **`updated_at` por trigger** | Un trigger de Postgres actualiza `updated_at` automáticamente en cada `UPDATE`, sin requerir lógica en la aplicación. |
