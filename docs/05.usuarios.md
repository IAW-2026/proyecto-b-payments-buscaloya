# 1.5 — Usuarios Compartidos

> **Tipo B — Plataforma de Delivery**

El sistema utiliza **Clerk** como servicio centralizado de autenticación. Los usuarios se autentican a través de Clerk independientemente de qué app estén usando, y la identidad se propaga entre servicios mediante el token JWT emitido por Clerk.

---

## ¿Qué apps comparten usuarios?

| Usuario (Rol) | Apps donde puede autenticarse |
|---------|------------------------------|
| **Comprador** (`buyer`) | Buyer App |
| **Vendedor / Restaurante** (`seller`) | Seller App |
| **Repartidor / Drone** (`courier`) | Delivery App |
| **Administrador Financiero** (`finance_admin`) | Payments App |
| **Administrador de Plataforma** (`system_admin`) | Todas las Apps (Buyer, Seller, Delivery, Payments) |

---

## Claims del JWT relevantes por app

| App | Claims utilizados | Para qué |
|-----|------------------|----------|
| Buyer App | `sub` (user ID), `role` | Identificar comprador, verificar rol `buyer` o `system_admin`. |
| Seller App | `sub` (user ID), `role` | Identificar vendedor, verificar rol `seller` o `system_admin`. |
| Delivery App | `sub` (user ID), `role` | Identificar repartidor logístico, verificar rol `courier` y habilitar su telemetría. Permitir a `system_admin` acceder al Centro de Comando (Admin). |
| Payments App | `sub` (user ID), `role` | El `sub` se almacena como `buyer_id` en la tabla `orders`. Se verifica rol `finance_admin` o `system_admin` para acceso administrativo. |

---

## Estrategia de roles

La plataforma utilizará **Clerk** como proveedor de identidad centralizado y fuente de verdad de la autenticación. La gestión de autorizaciones se manejará mediante el uso de la propiedad `publicMetadata` embebida en el perfil del usuario dentro de Clerk.

**Estructura de metadata en Clerk:**
```json
{
  "publicMetadata": {
    "role": "buyer" // Valores posibles: "buyer", "seller", "courier", "finance_admin", "system_admin"
  }
}
```

**Resolución en las aplicaciones (Ej. Delivery App):**
1. **Login:** El usuario inicia sesión a través del flujo provisto por Clerk.
2. **Validación (Middleware Edge):** El middleware de Next.js lee el JWT inyectado en las cookies o headers.
3. **Decisión de Acceso:**
   - Si el `role` es `courier`, se le muestra la interfaz táctica (HUD) para realizar misiones operativas.
   - Si el `role` es `system_admin`, se le permite el acceso a la "Capa 6: Centro de Comando", panel de control para supervisar toda la flota y auditar logs.
   - Si el `role` es distinto (ej. `buyer`), el middleware bloquea el acceso devolviendo un `403 Forbidden` y lo redirige a la landing page u otra app.

---

## Comunicación máquina a máquina (SERVICE_TOKEN)

Además del JWT de Clerk —que identifica usuarios humanos—, el sistema usa un `SERVICE_TOKEN` para las llamadas **backend-to-backend** donde no hay un usuario activo en la sesión (ej. Seller → Payments, Payments → Seller, Payments → Buyer).

Es una cadena secreta compartida entre todos los servicios, enviada como:
```
Authorization: Bearer <SERVICE_TOKEN>
```

> En producción se usaría un token distinto por par de servicios. Para este proyecto universitario con 4 servicios controlados, un secreto compartido es suficiente.

---

## Autenticación por endpoint — Payments App

| Endpoint | Caller | Mecanismo |
|----------|--------|-----------|
| `POST /payments/orders` | Seller App | SERVICE_TOKEN |
| `GET /checkout/{order_id}` | Buyer (browser) | JWT de Clerk (cookie de sesión) + validar `sub == order.buyer_id` |
| `POST /api/payments/process` | Buyer (browser, desde el Brick) | JWT de Clerk |
| `POST /api/payments/issue` | Seller App | SERVICE_TOKEN |
| `POST /api/payments/webhook` | MercadoPago | Público |
| `POST /payments/operations/{order_id}/close` | Delivery App | SERVICE_TOKEN |
| `GET /api/health` | Cualquiera | Público |

### Llamadas salientes de Payments

| Destino | Endpoint | Mecanismo |
|---------|----------|-----------|
| Seller App | `POST /orders/{order_id}/notify` | SERVICE_TOKEN |
| Seller App | `PATCH /orders/{order_id}/status` | SERVICE_TOKEN |
| Buyer App | `PATCH /api/purchases/{id}/status` | SERVICE_TOKEN |

---

## Middleware de Payments

`src/middleware.ts` exige `Authorization: Bearer <token>` en todas las rutas `/api/*` **excepto**:
- `/api/health` — público
- `/api/payments/webhook` — público (MercadoPago no envía token propio)

Las rutas `/checkout/*` quedan fuera del middleware de API; la autenticación se realiza con `auth()` de Clerk dentro del Server Component, validando además que `userId == order.buyer_id`.
